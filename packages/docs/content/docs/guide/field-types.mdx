`sz` mirrors `z`: anything you'd reach for in Zod has an `sz` equivalent that *also* knows
its SurrealQL type. Use it exactly like `z`.

## Scalars, formats, and composites

```ts twoslash
import { sz } from "surreal-zod";
import { z } from "zod";

sz.string();
sz.number();        // -> number
sz.int();           // -> int
sz.float();         // -> float
sz.bigint();
sz.boolean();

// String formats — those with a server validator bake an ASSERT (see below)
sz.email();         // -> string  ASSERT string::is_email($value)
sz.url();           // -> string  ASSERT string::is_url($value)

// Composites
sz.enum(["admin", "member"]);
sz.literal("active");
sz.union([sz.string(), sz.number()]);
sz.tuple([sz.number(), sz.number()]);
sz.array(sz.string());                 // or sz.string().array()
sz.object({ x: sz.number() });         // nested SurrealQL object
sz.record(z.string(), sz.number());    // open-keyed record<string, number> (key is a raw z schema)
```

## Surreal-native types

These map to SurrealDB's own datatypes and ride a codec so your app sees an ergonomic JS
value while the DB stores the native one.

```ts twoslash
import { sz } from "surreal-zod";
// ---cut---
sz.datetime();      // app: Date            <-> db: DateTime
sz.uuid();          // app: string          <-> db: Uuid     (native uuid type)
sz.duration();      // app/db: Duration
sz.decimal();       // app/db: Decimal      (arbitrary precision)
sz.bytes();         // app: Uint8Array      <-> db: bytes
sz.file();          // app/db: FileRef
sz.geometry();      // app/db: Geometry     (optionally sz.geometry("point"))
sz.recordId("user"); // -> record<user>
```

The codec is why a `datetime` field round-trips cleanly:

```ts twoslash
import { sz } from "surreal-zod";
// ---cut---
const at = sz.datetime();
const wire = at.encode(new Date());   // -> DateTime (the DB wire type)
//    ^?
const app = at.decode(wire);          // -> Date
//    ^?
```

## Field wrappers

Standard Zod wrappers are available and compose with the `$`-methods. Note that nullability
maps to distinct SurrealQL types:

```ts twoslash
import { sz } from "surreal-zod";
// ---cut---
sz.string().optional();   // -> option<string>
sz.string().nullable();   // -> string | null
sz.string().nullish();    // -> option<string | null>   (.optional().nullable())
sz.string().array();      // -> array<string>
sz.string().default("x"); // app-side Zod default (distinct from the DB-side $default)
```

<Callout type="info">
  `.optional()` / `.default()` are **app-side** Zod wrappers. To set a **database**
  `DEFAULT`/`VALUE`, use the `$`-methods (`$default`, `$value`) covered in
  [Generating DDL](/docs/guide/generating-ddl).
</Callout>

## Type mapping reference

| `sz`                        | SurrealQL type        | App type        |
| --------------------------- | --------------------- | --------------- |
| `sz.string()`               | `string`              | `string`        |
| `sz.int()` / `sz.float()`   | `int` / `float`       | `number`        |
| `sz.boolean()`              | `bool`                | `boolean`       |
| `sz.datetime()`             | `datetime`            | `Date`          |
| `sz.uuid()`                 | `uuid`                | `string`        |
| `sz.duration()`             | `duration`            | `Duration`      |
| `sz.decimal()`              | `decimal`             | `Decimal`       |
| `sz.bytes()`                | `bytes`               | `Uint8Array`    |
| `sz.recordId("t")`          | `record<t>`           | `RecordId`      |
| `sz.object({ … })`          | `object`              | `{ … }`         |
| `sz.array(x)`               | `array<x>`            | `x[]`           |
| `T.optional()`              | `option<T>`           | `T \| undefined`|
| `T.nullable()`              | `T \| null`           | `T \| null`     |

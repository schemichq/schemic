[Zod](https://zod.dev), and get three things from that single definition:

- **`decode` / `encode`** â€” map DB rows â‡„ app objects across Zod's two channels via

No code generation step, no separate schema language: your Zod schema *is* the schema.

## The idea

A field is a stock Zod schema plus a little SurrealQL metadata. The JSâ‡„DB mapping rides
on Zod's two native channels â€” the **encoded** side (`z.input`) is the DB wire type, the
**decoded** side (`z.output`) is your app type â€” so a `datetime` is a `DateTime` on the
wire and a `Date` in your code, for free.

```ts twoslash
import { sz, table, defineTable, type App } from "surreal-zod";
import { surql } from "surrealdb";

const User = table("user", {
  id: sz.string(),                                  // -> record<user>
  name: sz.string(),
  email: sz.email(),                                // -> ASSERT string::is_email($value)
  status: sz.string().$default("pending"),          // DB-side DEFAULT
  createdAt: sz.datetime().$default(surql`time::now()`).$readonly(),
});

// One definition, three outputs:
const ddl = defineTable(User);                      // SurrealQL DDL (a string)
type User = App<typeof User>;                        // the decoded app type
const payload = User.encode({ name: "Ada", email: "ada@example.com" });
//    ^?
```

That `email` field already knows it should emit `ASSERT string::is_email($value)`, `status`
is optional on create because the DB fills its default, and `createdAt` is a `Date` in
`App<User>` but a `DateTime` in the encoded payload â€” all from the one schema above.
